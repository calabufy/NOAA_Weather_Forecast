## P1 — высокий приоритет

## [x] [P1] Ретраи tenacity не действуют на стриминг бюллетеней (`http_iter_lines`)

**Расположение:** `app/sources/__init__.py`, функция `http_iter_lines`, строки 212–224;
потребители — `nbm.fetch_nbs_raw`, `mav.fetch_mav_raw`, `met.fetch_met_raw`.

**Проблема:**
`@retry` навешан на **генераторную функцию**. Вызов такой функции лишь создаёт объект
генератора и всегда «успешен» — tenacity видит успех и завершает работу. Реальные сетевые
операции (соединение, чтение 28 МБ NBS) происходят при итерации, уже **вне** retry-обёртки.
Итого: у самого тяжёлого и наиболее подверженного сбоям пути (bulk-файлы NBM/MAV/MET)
ретраев нет вообще, в отличие от `http_get_text`/`http_get_json`. Один сетевой сбой →
пропуск цикла модели → ложный алерт в чат (частично смягчается повторными запусками джоба
по расписанию, но внутрицикловые ретраи задекларированы и ожидаются).

**Текущее поведение:**
При успешной сети — корректный стриминг и вырезание блока станции. При сбое —
исключение сразу пробрасывается потребителю без повторных попыток. Сохранить: стриминг
(не загружать 28 МБ в память целиком), контракт `fetch_*_raw() -> str` (блок станции).

**Предлагаемое изменение:**
Переместить границу ретрая так, чтобы внутри неё выполнялась вся итерация. Самый простой
способ без изменения потребителей: объединить скачивание и вырезание блока в одну
ретраящуюся функцию, а `http_iter_lines` упразднить или сделать приватной деталью.

**Пример структуры решения:**
```python
# app/sources/__init__.py
@retry(reraise=True, stop=stop_after_attempt(4),
       wait=wait_exponential(multiplier=1, min=2, max=30))
def http_stream_extract(url: str, extractor: Callable[[Iterator[str]], str]) -> str:
    """Скачать URL стримингом и вернуть результат extractor'а по строкам.

    Вся итерация — внутри ретрая: сбой соединения/чтения повторяется целиком.
    """
    with httpx.Client(...) as client:
        with client.stream("GET", url) as r:
            r.raise_for_status()
            return extractor(r.iter_lines())

# app/sources/nbm.py
def fetch_nbs_raw(run_date, cycle):
    url = ...
    return http_stream_extract(url, extract_station_block)
```

**Ожидаемый эффект:** сетевые сбои при скачивании бюллетеней реально ретраятся;
поведение кода совпадает с задокументированным.

**Риски:**
Повторная попытка теперь заново скачивает файл с начала — это и есть желаемое поведение,
но увеличивает трафик при флапающей сети (приемлемо: 4 попытки максимум).
`extract_station_block` бросает `ParseError`, если блок не найден, — ретраить это
бессмысленно; добавить `retry=retry_if_exception_type(httpx.HTTPError)` либо принять
лишние повторы как безвредные (блок либо есть, либо нет — 4 холостых повтора при ParseError).

**Проверка результата:**
Существующие тесты парсеров зелёные; новый юнит-тест: мок-стрим, падающий на первой
итерации, — убедиться, что происходит повторная попытка (сейчас — не происходит).

**Зависимости:** нет.

**Оценка сложности:** низкая.

## [x] [P2] N+1-запросы в `error_series` и повторные выборки в `/errors`

**Расположение:** `app/db/repo.py`, функция `error_series`, строки 198–212;
`app/bot/handlers.py`, `cmd_errors`, строки 72–91.

**Проблема:**
`error_series` для каждого дня из `list_actuals` делает отдельный SQL-запрос
`official_forecast` (это ровно тот паттерн «выбор последней записи по идентификатору
в цикле», который стоит заменить оконной функцией). `cmd_errors` вызывает `error_series`
для каждой из 3 моделей — `list_actuals` за один и тот же диапазон выполняется трижды.
Итого до ~1100 запросов на команду `/errors` при годовом окне.
**Честная оговорка:** SQLite внутрипроцессный, объёмы — сотни строк; это миллисекунды,
и мотивация здесь — простота и единый SQL-инвариант, а не измеримое ускорение. Заявлять
ускорение без профилирования нельзя. Заодно уменьшится время блокирующего чтения прямо
в event-loop бота (в `cmd_errors` чтение выполняется без `to_thread` — сейчас допустимо,
но чем меньше работы в loop, тем лучше).

**Текущее поведение:**
Для каждой даты с фактом берётся «зачётный» прогноз: последний цикл модели строго до
local midnight target-даты (полночь считается в Python через `timeutil.local_day_bounds`,
DST-корректно). Дни без прогноза или факта исключаются. Это поведение — контракт,
менять нельзя.

**Предлагаемое изменение:**
Один запрос на модель с `ROW_NUMBER()` (SQLite ≥ 3.25 поддерживает оконные функции).
Ключевая тонкость: отсечка `cycle < local_midnight_utc` **своя для каждой даты** и
считается в Python (DST!). Не переносить расчёт полуночи в SQL — вместо этого передать
пары (дата, отсечка) как CTE `VALUES`:

```python
def error_series(conn, model, start, end):
    days = list_actuals(conn, start, end)          # один раз, снаружи цикла
    cutoffs = [(_fmt_date(a.date), _fmt_cycle(timeutil.local_day_bounds(a.date)[0]))
               for a in days]
    # WITH cutoff(target_date, midnight) AS (VALUES (?,?),(?,?),...)
    # SELECT ... FROM forecasts f JOIN cutoff c USING(target_date)
    # WHERE f.model = ? AND f.cycle < c.midnight
    # QUALIFY-эквивалент: ROW_NUMBER() OVER (PARTITION BY f.target_date
    #                                        ORDER BY f.cycle DESC) = 1
```

Альтернатива с тем же эффектом и меньшим SQL: оставить `error_series` как есть, но
загрузить **все** подходящие прогнозы модели за диапазон одним запросом и выбрать
«последний цикл до отсечки» в Python по уже отсортированному списку. Выбрать вариант,
который окажется читабельнее; не смешивать оба.

В `cmd_errors` — вычислить `list_actuals` один раз и передавать в `error_series`
(или новую сигнатуру `error_series(conn, model, actuals)`), чтобы не читать факты трижды.

**Ожидаемый эффект:** 3 запроса вместо ~1100; правило «зачётного» прогноза выражено
в одном месте; меньше работы в event-loop.

**Риски:**
Главный: тихое изменение правила отсечки (строгое `<`, полночь по LA с DST). Граничные
случаи — цикл ровно в полночь (исключается), переходы DST в марте/ноябре.

**Проверка результата:**
Перед изменением снять эталон: для текущей БД `data/la_weather.db` выгрузить результат
`error_series` по всем моделям за максимальный диапазон (CSV/список кортежей). После —
сравнить поэлементно (даты, значения прогнозов и фактов, порядок). Существующие тесты
`test_repo.py` (в т.ч. `test_official_forecast_picks_latest_cycle_before_local_midnight`)
должны остаться зелёными; добавить тест на `error_series` с циклом ровно в полночь.

**Зависимости:** нет.

**Оценка сложности:** средняя.

## [x] [P2] `/forecast`: при сбое забора пользователь не получает ничего

**Расположение:** `app/bot/handlers.py`, `cmd_forecast`, строки 58–69.

**Проблема:**
Если `live.forecast_tomorrow()` бросит исключение (тотальный сбой сети до истечения
ретраев), `finally` удалит плашку «Загрузка прогноза...», исключение уйдёт в aiogram —
пользователь останется без какого-либо ответа. Кроме того, `await loading.delete()`
в `finally` сам может бросить (например, при гонке с Telegram API) и замаскировать
исходное исключение.

**Текущее поведение:**
Частичные сбои уже обрабатываются внутри `live` (модель → «—»); необработанным остаётся
только тотальный отказ. Формат успешного ответа не меняется.

**Предлагаемое изменение:**
```python
loading = await message.answer("Загрузка прогноза...")
try:
    target, points = await live.forecast_tomorrow()
except Exception:
    log.exception("сбой /forecast")
    await message.answer("Не удалось получить прогноз, попробуйте позже.")
    return
finally:
    with suppress(Exception):   # delete не должен маскировать исход
        await loading.delete()
await message.answer(formatting.format_forecast(target, points))
```

**Ожидаемый эффект:** честное сообщение вместо молчания; ошибка удаления плашки
не маскирует основную.

**Риски:** нет (успешный путь байт-в-байт тот же).
